You have two passwords, a long cryptographically-strong one that you'll probably never manage to memorize, and a short PIN that is used locally to decrypt the long one on your device. Content is encrypted with 128-bit RC4, generally considered safe but here is some informed discussion of an attack from March 2013. This attack appears to be useful only for huge amounts of data encrypted with the same key, so inapplicable here. I'm considering switching to Rijndael in cipher-feedback mode.

Cryptographic rules of engagement

  • The password is stored in PIN-encrypted form in the browser's localStorage.
  • The content is stored in PW-encrypted form in the browser's localStorage and in the server's database.
  • If the PIN is incorrectly guessed three times, localStorage is wiped.
  • A salted hash of the password (not the password itself) is stored in the server's database.
  • The salted hash never leaves the server.
  • The PIN never leaves the browser, and is never stored in localStorage.
  • The plaintext password is never stored in localStorage or on the server.
  • The server will see the unencrypted password only when the user registers or logs in. The user can see his own unencrypted password by typing his PIN into the window on the About page.
  • When the password is successfully decrypted, the PIN input field is erased. The PW is never visible in the browser, except when entering it on the login page, or retrieving it on the About page.
